# AI Engineering Notes — K-Tactics Lab 2026

이 문서는 K-Tactics Lab의 AI/데이터 관련 **설계 결정과 그 근거**를 정리한다. 목표는 두 가지다. (1) 심사자와 리뷰어가 "무엇을, 왜 그렇게" 만들었는지 검증할 수 있게 하고, (2) LLM을 "붙였다"가 아니라 **시스템으로 설계**했음을 보인다.

## 0. 핵심 원칙

> **LLM 토큰은 "예측 불가한 입력에 대한 열린 자연어 생성"이 진짜 가치인 곳에만, 이벤트 기반으로 상한을 두고 쓴다. 타이머나 고빈도 반복에는 쓰지 않는다.**

이 원칙이 아래 모든 결정을 지배한다. 특히 "무엇을 AI로 하지 **않을지**"가 더 중요하다.

## 1. 비용 아키텍처 (3-티어)

| 티어 | 대상 | 런타임 비용 |
|---|---|---|
| 오프라인 (1회) | 데이터 파이프라인(`parse_stats.py`), 여론 댓글 뱅크 생성(`gen_fan_comments.py`) | 배포 후 $0 |
| 런타임 로컬 | 몬테카를로 경기 시뮬, 여론 댓글 상태조건 선택, 결정론적 전술 피드백 | $0 |
| 런타임 LLM | Coach V 채팅/분석 + AI 상대감독 (사용자 액션 시에만) | 무료 티어 기준 $0, 유료 전환 시 콜당 센트 이하 |

설계 의도: 비싼 생성은 **개발 중 1회** 수행해 정적 자산으로 굳히고(offline generation), 런타임에는 상태로 **고르기만** 한다(runtime conditioning). 정말로 열린 대화가 필요한 Coach V만 라이브 LLM을 쓰되 사용자 액션에만 발화한다.

## 2. 모델 선택 매트릭스

| 표면 | 모델 | 근거 |
|---|---|---|
| 경기 결과 시뮬 | **모델 없음 (로컬 Poisson/몬테카를로)** | 득점은 Poisson으로 잘 근사된다. 선수 레이팅에서 λ를 유도하면 결정론적·검증가능·$0. LLM보다 정확하고 설명 가능. |
| Coach V 채팅 | **Gemini 2.5 Flash-Lite (무료)** | 짧은 그라운디드 답변에 충분. 저지연·무료. 이벤트 기반(질문 시). |
| 사전분석 / AI 상대감독 | **Gemini 2.5 Flash-Lite (무료)** | 저빈도 호출. 구조화 출력은 `responseSchema`(JSON 모드)로 형식 강제. |
| 여론 댓글 | **오프라인 배치 생성 → 런타임 $0** | 타이머(4.5초)로 발생하므로 콜당 LLM은 비용 함정. 상태 태깅 뱅크에서 선택. |

### 열린 항목 (재검토 예정)
Flash-Lite 무료 티어는 두 리스크가 있다. (a) 무료 rate-limit 소진 시 심사 중 폴백 노출, (b) 전술 추론/JSON 신뢰성이 상위 모델보다 약함. 이를 대비해 **LLM 호출을 `callLLM()` 한 함수 뒤로 추상화**했다(`api/coach.js`). 대회 임박 시 심사-크리티컬 경로(상대감독)만 유료·고신뢰 모델(예: Claude Haiku)로 승격하는 것이 이 함수 body 교체만으로 가능하다.

## 3. 데이터 파이프라인 (`scripts/parse_stats.py`)

- **출처**: FBref, Korea Republic, World Cup 2026 (Standard / Shooting / Misc 스탯).
- **스코프**: 실제 출전(> 0 MP)한 **20인만** 포함. 미출전 선수는 제외해 표본을 명확히 한다.
- **변환**: per-90 원자료(`s90`, `sh`, `sot`, `int`, `tkl`, `crs` 등)를 포지션 베이스 테이블(`ATK_BASE`/`DEF_BASE`/`MID_BASE`) + 가중식으로 4대 능력치(공격/수비/중원/체력, 30~96)로 변환한다.
  - 표본 가중치 `w = min(1, s90/K)`로 소표본(적은 출전 시간)의 변동을 억제한다.
  - **GK 분기**: 골키퍼 수비력은 세이브율(`save_pct`) 기반 별도 공식.
  - **스태미나**: 나이 감쇠 모델(`85 - (age-27)*0.6 ...`).
  - **수동 보정**: 김민재는 3경기 표본의 태클/인터셉트 수치가 실제 기량을 과소평가해 `override_def`로 보정하고 코드에 `[수동 보정]`으로 명시한다.
- **소표본 주의**: 조규성(31분), 양현준(20분) 등은 `statStr`에 "(소표본)"으로 표기해 과신을 막는다.
- **정직성 정제**: 초기에 미측정 지표 `npxG/progPasses/aerialWon`을 `'─'` 플레이스홀더로 두었으나, **표시하지 않는 데이터를 지표인 척 두지 않기 위해 제거**했다. 스카우팅 모달의 해당 "연동 대기" 자리도 실제 파생 능력치(공/수/중/체력, 슈팅, 침착성 + FBref 캡션)로 교체했다.

## 4. 경기 결과 모델 (로컬 Poisson 몬테카를로)

`app.js`의 `secondHalfLambdas` / `poissonSample` / `runMonteCarlo`.

- 후반전 양팀 득점을 **Poisson(λ)**에서 1,000회 샘플링해 하프타임 스코어에 더하고, 승/무/패 분포와 최빈 스코어를 집계한다.
- **λ 유도**: `λ_kor = 0.72 · (공격력 / 상대수비) · 중원계수`, `λ_opp = 0.72 · (상대공격 / 수비력) / 중원계수`. 여기에 전술 다이얼(압박/성향/루트), 후반 체력 감소, 실행된 교체(프레시 레그), 그리고 **AI 상대감독의 카운터 전술**(`opponentModifiers`)이 곱셈 보정으로 반영된다.
- **검증**(실제 흐름 기준, 1,000회):
  - 강한 한국 + 고압박·하프스페이스(하프타임 1:0): 승 ~89%
  - 손흥민 벤치 + 소극적(하프타임 0:1): 승 ~8%, 패 ~68%
- 전술 셋업이 결과를 인과적으로 좌우하므로 "내가 감독이다" 몰입과 정합한다. 기존의 `Math.random()` 2회 + "1,000회 몬테카를로"라는 허위 표기를 실제 1,000회 연산으로 대체했다.

## 5. AI 상대 감독 (에이전틱, 적대적)

`fetchOpponentPlan` / `scriptedCounterPlan` / `opponentModifiers`.

- 킥오프 시 LLM이 우리 XI·스탯을 스카우팅하고 **카운터 전술을 구조화 JSON**(`counterFormation`, `counterDials`, `reasoning`)으로 반환한다. Gemini `responseSchema`로 허용 옵션(enum)을 강제한다.
- 반환된 카운터는 시뮬의 λ에 반영되어 실제로 결과를 바꾼다(적대적 루프). 예: 상대가 고압박을 고르면 우리 득점 기대가 낮아진다.
- **폴백**: LLM 응답 전/실패 시 `scriptedCounterPlan()`이 우리 강점을 지우고 약점을 노리는 휴리스틱 카운터를 동기적으로 즉시 세팅한다. 시뮬은 항상 유효한 상대 계획을 갖는다.

## 6. 여론 댓글 (offline generation + runtime conditioning)

- `data/fan_comments_2026.js`: 경기 상태로 **태깅된** 댓글 뱅크(정적 자산). 태그 예: `sonBenched`, `nopassback`, `highPress`, `tenback`, `fatigue`, `strongAttack`, `pre/half/full`.
- 런타임 `activeFanTags(state)`가 라이브 상태에서 활성 태그를 뽑고 `pickFanComment()`가 매칭 후보에서 선택한다(60% 상황 매칭, 그 외 일반). LLM 호출 0회.
- `scripts/gen_fan_comments.py`(오프라인, 선택)로 뱅크를 확장·갱신한다. 이 스크립트만 LLM을 쓰며 런타임에는 관여하지 않는다.
- 감정 앵커: 실제 2026 월드컵 남아공전(비기면 통과인데 1-0 패), 손흥민 벤치, 홍명보 전술 부재, U자 백패스 논란을 댓글에 녹였다.

## 7. 서버리스 프록시 & 프로덕션 (`api/coach.js`, `vercel.json`)

- **왜 프록시인가**: 클라이언트 직접 호출은 API 키를 노출하고 대회 규칙(심사자 키 불필요)에 위배된다. 프레임워크(LangChain 등)는 zero-build 레포에 불필요한 의존성. 그래서 Vercel 서버리스 함수에서 **raw `fetch`**로 호출하고 키는 `process.env.GEMINI_API_KEY`(서버 전용)에만 둔다.
- **계약**: `POST /api/coach` body `{ mode, message?, state }` → `{ reply, model, fallback, opponent? }`.
- **프롬프트 설계**: `system_instruction`에 페르소나 + 라이브 보드 상태(상대 브리핑·다이얼·스탯·라인업)를 주입해 그라운딩한다. 구조화가 필요한 opponent 모드는 `responseSchema`로 JSON을 강제하고 클라이언트는 방어적으로 파싱한다.
- **비용/남용 방어**: 무료 티어라 비용 ≈ $0. 그 위에 per-IP rate-limit + 출력 토큰 상한 + 입력 길이 클램프로 무료 한도 보호·남용 차단.
- **Graceful degradation**: 키 미설정/무료 한도/네트워크 실패는 전부 `fallback: true`로 응답하고, 클라이언트는 스크립트 폴백을 쓴다. 앱은 `index.html` 더블클릭(`file://`)으로도 항상 열리며 AI만 폴백된다(오프라인 심사 대비).
- **데이터 사용 약관**: Gemini 무료 티어는 입력을 모델 개선에 사용할 수 있다. 본 서비스는 민감정보를 다루지 않으므로 허용 범위로 판단한다.

## 8. 요약: 이 프로젝트가 보이려는 AI 엔지니어링 역량

- 컨텍스트 엔지니어링(스탯/상태 그라운딩), 구조화 출력(responseSchema + 방어적 파싱)
- 적대적 에이전틱 루프(AI 상대 감독이 사용자 전술에 반응)
- 정량 모델링(Poisson 몬테카를로, 검증된 확률분포)
- 비용 아키텍처(offline generation + runtime conditioning, 이벤트 기반 LLM)
- 프로덕션 감각(키 은닉, rate-limit, graceful degradation, 프로바이더 추상화)
